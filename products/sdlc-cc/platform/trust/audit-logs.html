Audit-log architecture

Same shape across sdlc-platform · AMLIQ · Claw. Last reviewed 2026-05-20.

Row shape

Every auditable event — auth, admin action, sensitive data mutation, DLP detection, billing change — produces one row:

{
  "id":         "ulid",
  "tenant_id":  "uuid",
  "actor_type": "user | service | system",
  "actor_id":   "uuid | null",
  "action":     "domain.verb",      // e.g. dlp.inbound, rbac.grant
  "target_type":"http_request | tool_use | document | user | tenant | …",
  "target_id":  "string",
  "before":     { … }  | null,
  "after":      { … },
  "created_at": "RFC3339",
  "hmac":       "hex"                // chained over (prev_hmac || row)
}

HMAC chain

  • Each row's hmac is computed over the previous row's hmac concatenated with the canonical JSON of the current row.
  • The chain key is per-tenant and rotatable; rotation creates an audit row of its own (action: audit.key_rotate).
  • Reading the log verifies the chain end-to-end; a tampered or removed row breaks verification and produces a console-visible warning + a Prometheus counter increment.

Replay semantics

  • Any past detection event can be replayed against the current preset set without re-touching the original prompt text. This means: change a preset → reprocess the audit log → see whether a previously-allowed call would be redacted today.
  • Replay output is itself an audit row (action: audit.replay), so a what-if exercise is also auditable.

Retention

Default 13 months. Configurable to 24/36/60 months on Enterprise. We do not delete audit rows on a tenant's request without a documented business reason — that itself produces an audit row.

Reading the log

  • GET /v1/me/redactions — every DLP detection the caller's user id appears in (GDPR Art. 15).
  • GET /v1/audit/events — admin-only, full tenant scope (RBAC: audit:read).
