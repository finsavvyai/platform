## Gaps — work units for the grinder

Each gap below is **one hour of work** or less and writes into the
runtime path (no primitives). Pick the highest-priority one that

### G1 — Bicep coverage 52% (12/23 detected)

**Gap.** GraphQL primaryLanguage says 23 Norlys repos are Bicep-
primary. `pushci init` detects only 12. Half the Azure IaC fleet is
classified as `?` or rolled into another stack.

**Hypothesis.** `internal/detect/markers.go` `*.bicep` glob fires on
file presence but a higher-priority marker (e.g. `Dockerfile`,
`*.csproj`, or `.github/workflows/*.yml`) wins the stack pick when
present in the same repo. Many Norlys Bicep repos ship a Dockerfile
for the linter/validator container — that pulls them into `docker`.

**Fix shape.** Bicep should win over Docker when `*.bicep` count
≥ Dockerfile count and no compiled-language marker present.
Mirroring the Terraform-vs-Docker rule already in `pickStack`.

**Verify.** Add fixture in `internal/detect/bicep_dogfood_test.go`:
write `main.bicep`, `Dockerfile`, `.github/workflows/ci.yml`, expect
`Stack == Bicep`. Ground-truth reference: any of the
`cloud-platform-*` repos in the dogfood report (12/12 Bicep
detected) vs `edm-*` repos (4 Bicep listed by GraphQL, varied
detection).

### G2 — Kotlin standalone never returned

**Gap.** `Stack.Kotlin = "kotlin"` constant exists in stack.go:28
but the JVM detection rolls every Kotlin repo into `Stack.Java`
with framework `kotlin`. Reports show 0 Kotlin rows; GraphQL counts
6.

**Why it matters.** Three of the six Kotlin repos are the customer-
visible `norlys-play-*` family (Android/KMM/Samsung/Browser) — the
strongest pilot scope option per the Norlys fleet strategy memo.

**Fix shape.** When `looksLikeKotlinDsl` (java_gradle_parse.go:28)
returns true at the project root AND `*.java` count is zero, return
`Stack.Kotlin`. Java + Kotlin polyglot still returns `Stack.Java`
with framework `kotlin` — that's correct.

**Verify.** Add `internal/detect/kotlin_standalone_test.go` with a
fixture that writes `build.gradle.kts` + `settings.gradle.kts` +
`src/main/kotlin/App.kt` (no `*.java`). Expect `Stack == Kotlin`.

### G3 — 82 repos return `?` with empty pushci.yml

**Gap.** A quarter of the fleet generates `0 B` output and a
trailing "no supported stack detected" warning that does not
produce a pushci.yml at all. Many of these are config / docs /
template / org-policy repos for which a no-op pipeline is the
right answer — but the silent 0 B file is a worse UX than emitting
a stub.

**Fix shape.** When `detect.Scan` returns zero projects, write a
stub `pushci.yml`:

```yaml
"on":
  - push
  - pull_request
# pushci could not detect a build stack in this repo.
# If you intend CI here, add a Dockerfile, package.json,
# pyproject.toml, *.csproj, or other recognised marker. See:
# https://pushci.dev/docs/detect
```

**Verify.** Ground-truth: `cisco-nso`, `community`, `BuildInfra`,
`developer-helper-tools` from the dogfood report — all `0 B` today.
After fix, all four should have a pushci.yml of ~200 B.

### G4 — Docker-only emits 36 B "on:" only

**Gap.** Three repos (`.github`, `api-management-pipelines`,
`uc-sh-tp-base-images`) detect as Docker but emit only the
`on: [push, pull_request]` triggers. Should emit at least a
`docker build .` stage so the pipeline does something.

**Fix shape.** In `cmd_init_generate.go:pickStages`, when the only
detected project is `Stack.Docker` and a Dockerfile lives at root,
emit a single `build` stage with `docker build -t $IMAGE .` (or
`docker buildx build` if `buildx.toml` present).

**Verify.** Add `cmd/pushci/cmd_init_docker_only_test.go` with a
fixture that writes only a Dockerfile. Expect 1 stage named `build`
with the `docker build` command.

### G5 — Migration drops to 0 stages (already partly fixed)

**Gap.** pos-dk shipped a fix in commit `dfb532f` for the case
where the GH Actions importer maps zero steps. The same pattern may
exist for other generic CI sources where the importer succeeds at
parse but produces empty stages.

**Status.** Done for GH Actions. Audit remaining `FromGenericCI`
kinds (`gitlab`, `circleci`, `bitbucket`, `azure`) — confirm they
either succeed with stages OR fall through to heuristic. The
recent grinder commit `303fddb` (GitLab `!reference` inlining)
suggests GitLab path was already exercised.

**Verify.** Grep for `mergeSecondaryHeuristic` callers, walk each
`FromGenericCI*` flag through the importer to confirm zero-stages
case can't slip through silently.

## Launch readiness — viral / Product Hunt + Show HN

These are gaps that would bite at a public launch. Same prioritisation
rules as G1–G6 above: prefer items the grinder can ship without an
external dep. Decisions / system-level items are flagged for a human.

### L1 — Init silent-failure stub fallback (CODE)

Same scope as G3 above. When `detect.Scan` returns zero projects,
write a stub `pushci.yml` instead of refusing. HN/PH crowd dogfoods
weird repos — silent 0 B output reads as "broken on first run."

### L2 — Migration importers thin past GH Actions (CODE)

Recent commits `303fddb` / `3681dd0` / `c8fa27e` / `be79b30` are all
GitLab importer fixes. CircleCI / Bitbucket / Azure DevOps importers
(`internal/migrate/`) are equally under-tested. Build the same
adversarial fixture matrix used for GitLab and apply.

### L3 — Pre-existing flaky tests in `internal/mcp` and `internal/security` (CODE)

Public CI shows red on launch day. CLAUDE.md treats these as
known-flaky; that's not viable when strangers read the README. Either
fix the underlying race / fixture issue, or `t.Skip` with a tracked
issue link so the badge is green.

### L4 — TestInitAndRun is network-dependent (CODE)

Needs `npx` + network → flakes in hermetic CI and on first-install
demos. Either move to a stub `npx` shim or split into `_integration_test.go`
gated on `PUSHCI_NETWORK_TESTS=1`.

### L5 — Pin Go toolchain in `go.mod` ✓ SHIPPED 2026-05-08

Commit `7f21bdf` pinned `toolchain go1.26.3` to clear the two
stdlib CVEs (GO-2026-4971, GO-2026-4918). Local contributors no
longer get random govulncheck rejections after stdlib CVE drops.
Bump on each Go release. Left here for traceability.

### L6 — Docker-only repos emit 36 B (CODE)

Same scope as G4 above. Promote to launch-readiness because Docker is
the most common "first-thing-they-try" shape on HN.

### L7 — Bicep 52% / Kotlin 0% detection (CODE)

G1 + G2 above, promoted because Azure shops and Android shops are real
HN audiences and the clear-miss reads as "not for me."

### L8 — Cold-install proof recordings (DECISION + ASSET)

90-sec recordings of `npm i -g pushci` working in a fresh Cursor /
Codespaces / Claude Code sandbox. Retires the v1.4.2 sandbox-bug
narrative. Not the grinder's job — needs an asset producer.

### L9 — Windows install path (DECISION)

No `winget` / `scoop`. Half the dev market hits a curl-only install
that won't run. Either ship Windows paths or scope an explicit
"macOS / Linux / WSL only at v1.x" disclaimer on landing.

### L10 — Free AI tier without BYO-key (DECISION + CODE)

`internal/ai/claw_client.go` already implements the PushCI proxy.
Wire a rate-limited free path so first-run AI features work without
the user setting `ANTHROPIC_API_KEY` or similar. First-run AI is the
demo that makes the cost pitch land.

### L11 — Opt-in anonymous telemetry (DECISION + CODE)

5000 PH visitors install → you don't know who succeeded, where they
dropped, what stack broke. Opt-in anonymous "init result + stack
name" ping. Without this you fly blind post-launch.

### L12 — GH Actions release workflow + CF Pages auto-deploy (DECISION)

Both currently manual (release.yml blocked on org billing since
v1.4.0; CF Pages requires `wrangler pages deploy` by hand). Launch-
day hotfix path is fragile. Either fix billing or move release to a
different runner; wire CF Pages to the GitHub Actions push.
